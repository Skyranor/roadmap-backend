# Неделя 1: HTTP и Сетевые протоколы (Подробный конспект для Backend-инженера)

## 1. Введение: Модель OSI (Open Systems Interconnection — Эталонная модель взаимодействия открытых систем)

Модель OSI — это теоретическая модель, которая описывает, как данные передаются по сети от одного приложения к другому. Backend-инженеру не обязательно помнить все 7 уровней наизусть, но критически важно различать L4 и L7:

*   **L4 (Транспортный уровень / Transport Layer):** Отвечает за доставку пакетов данных между устройствами. На этом уровне мы работаем с **IP** (Internet Protocol — Интернет-протокол) адресами и **портами** (например, порт 80 или 443). L4 ничего не знает о содержимом пакетов (будь то JSON, HTML или картинка).
    *   *Примеры:* **TCP** (Transmission Control Protocol), **UDP** (User Datagram Protocol).
*   **L7 (Прикладной уровень / Application Layer):** Отвечает за логику самого приложения. На этом уровне данные уже собраны в осмысленный текст или формат.
    *   *Примеры:* **HTTP** (HyperText Transfer Protocol), **WebSocket**, **SMTP** (Simple Mail Transfer Protocol).

---

## 2. Разрешение имен: DNS (Domain Name System — Система доменных имён)

Компьютеры общаются только по числовым **IP-адресам** (например, `192.168.1.1` для IPv4 или длинным строкам для IPv6). Людям удобнее использовать буквенные имена (например, `api.omnia.com`).
**DNS** — это гигантская распределенная база данных (адресная книга), которая переводит удобочитаемое доменное имя в IP-адрес.

**Процесс (DNS Resolution):**
1.  Приложение запрашивает IP для `api.omnia.com`.
2.  Операционная система смотрит в локальный кэш.
3.  Если нет — запрос идет к DNS-серверу вашего провайдера (ISP — Internet Service Provider).
4.  Если и там нет, запрос идет по цепочке корневых серверов до тех пор, пока не будет найден нужный IP.

## 3. Транспорт: TCP против UDP

Перед тем как отправить HTTP-запрос, нужно выбрать способ доставки (L4).

### TCP (Transmission Control Protocol — Протокол управления передачей)
Это протокол с гарантией доставки.
*   **Особенности:**
    *   **Предварительная установка соединения (Handshake):** Прежде чем отправить первый байт данных, клиент и сервер обмениваются тремя пустыми пакетами: `SYN` (Synchronize) → `SYN-ACK` (Synchronize-Acknowledge) → `ACK` (Acknowledge). Это называется **3-way handshake** (тройное рукопожатие).
    *   **Гарантия доставки:** Если часть данных потерялась по пути, TCP автоматически запросит их повторную отправку.
    *   **Гарантия порядка:** Данные собираются на сервере строго в том порядке, в котором были отправлены.
*   **Использование:** HTTP/HTTPS, базы данных (PostgreSQL), SSH. Везде, где потеря одного байта критична.

### UDP (User Datagram Protocol — Протокол пользовательских датаграмм)
Это протокол отправки без гарантий (fire-and-forget — "выстрелил и забыл").
*   **Особенности:**
    *   Не устанавливает соединение (нет Handshake). Начинает отправку данных сразу.
    *   Не гарантирует доставку (если пакет потерялся — ну и ладно).
    *   Не гарантирует порядок пакетов.
*   **Использование:** Видеозвонки (Zoom/WebRTC), онлайн-игры, DNS-запросы. Везде, где скорость важнее 100% точности.

---

## 4. Безопасность: HTTPS, SSL и TLS

**HTTP** (HyperText Transfer Protocol) передает данные в открытом виде. Если клиент отправляет пароль, любой узел (маршрутизатор) на пути от клиента до сервера может прочитать этот пароль (атака Man-in-the-Middle — "человек посередине").

Для защиты используется **HTTPS** (HyperText Transfer Protocol Secure — Безопасный HTTP). Он оборачивает HTTP-трафик в криптографический туннель.

*   **SSL (Secure Sockets Layer — Уровень защищенных сокетов):** Старый, уязвимый и устаревший протокол шифрования. Сейчас не используется, но аббревиатура осталась в обиходе (например, "купить SSL-сертификат").
*   **TLS (Transport Layer Security — Безопасность транспортного уровня):** Современный протокол, пришедший на смену SSL.

**TLS Handshake (Рукопожатие TLS):**
После того как установлено TCP-соединение, стороны должны договориться о шифровании:
1.  Клиент шлет `ClientHello` (какие алгоритмы шифрования он поддерживает).
2.  Сервер отвечает `ServerHello` + присылает свой **SSL/TLS Сертификат** (свой цифровой паспорт, заверенный доверенным центром — Certificate Authority).
3.  Клиент проверяет сертификат и стороны генерируют общий секретный ключ (Symmetric Key), которым будут шифроваться все последующие HTTP-данные.

## 5. Полный жизненный цикл запроса (От браузера до Node.js сервера)

Когда клиент делает запрос `fetch('https://api.omnia.com/users')`:

1.  **DNS Resolution:** Перевод `api.omnia.com` в IP-адрес (например, 104.20.5.1). Занимает ~20-50 миллисекунд (мс).
2.  **TCP Handshake (L4):** Установка соединения по порту 443. Занимает 1 **RTT** (Round Trip Time — время, за которое пакет доходит до сервера и возвращается обратно).
3.  **TLS Handshake (L7/Security):** Обмен сертификатами и генерация ключей. Занимает еще 1-2 RTT.
4.  **HTTP Request (L7):** Браузер отправляет зашифрованное письмо (Метод GET, URL /users, заголовки). Занимает 0.5 RTT.
5.  **Server processing:** Ваш Node.js сервер получает запрос, идет в PostgreSQL, формирует JSON.
6.  **HTTP Response (L7):** Сервер отправляет JSON обратно клиенту. Занимает 0.5 RTT.

*Итого, на холодный (первый) запрос уходит 3-4 RTT, прежде чем фронтенд получит данные.*

## 6. Оптимизация: HTTP Keep-Alive

Так как установка соединения (TCP + TLS) очень "дорогая" по времени, умные инженеры придумали переиспользовать открытые соединения.

**Keep-Alive (Удержание соединения):**
По умолчанию в современных версиях HTTP (HTTP/1.1) включен заголовок `Connection: keep-alive`. Это означает, что после отправки ответа сервер **не закрывает** TCP-сокет. Если фронтенд сделает второй запрос к этому же серверу через миллисекунду, запрос полетит в уже открытый канал (шаги 1, 2 и 3 из списка выше пропускаются!).

> **Backend-совет:** Когда ваш Node.js сервер делает запросы к сторонним сервисам (например, к API OpenAI), по умолчанию встроенный `fetch` или `axios` могут закрывать соединения. Обязательно убедитесь, что вы используете HTTP/HTTPS Agent с включенным `keepAlive: true`, иначе ваш сервер будет тратить процессорное время и сеть на бесконечные TLS Handshakes.

## 7. Анатомия HTTP-запроса и ответа

HTTP — это **текстовый протокол**. Запрос и ответ — это просто строки текста, которые летят по TCP-сокету. Это можно увидеть на практике через `net.createServer()` — сырые байты после `.toString()` превращаются в читаемый текст.

### Структура HTTP-запроса

Запрос состоит из трёх частей, разделённых `\r\n`:

```
POST /api/workspaces/42/agents?status=active HTTP/1.1\r\n   ← Start line
Host: api.omnia.com\r\n                                     ← Headers
Authorization: Bearer eyJhbGciOi...\r\n
Accept: application/json\r\n
Content-Type: application/json\r\n
\r\n                                                         ← Пустая строка (разделитель!)
{"name": "Sales Bot"}                                       ← Body
```

| Часть | Что в ней | Пример |
|---|---|---|
| **Start line** | Метод + URL + версия HTTP | `GET /users HTTP/1.1` |
| **Headers** | key: value пары (метаданные) | `Content-Type: application/json` |
| **Body** | Тело (только для POST/PUT/PATCH) | `{"name": "Alex"}` |

Между headers и body — **пустая строка** (`\r\n\r\n`). Без неё клиент/сервер не поймёт, где заканчиваются заголовки.

### Структура HTTP-ответа

```
HTTP/1.1 201 Created\r\n                                    ← Status line
Content-Type: application/json; charset=utf-8\r\n           ← Headers
Location: /api/agents/73\r\n
X-Request-Id: req_a1b2c3\r\n
\r\n                                                        ← Пустая строка
{"id": 73, "name": "Sales Bot", "status": "draft"}         ← Body
```

### Парсинг запроса руками (как работает Express под капотом)

```javascript
const raw = data.toString()

// 1. Разделить headers и body по пустой строке
const [head, ...bodyParts] = raw.split('\r\n\r\n')
const body = bodyParts.join('\r\n\r\n')

// 2. Распарсить start line
const lines = head.split('\r\n')
const [method, url, httpVersion] = lines[0].split(' ')

// 3. Распарсить заголовки
const headers = {}
for (let i = 1; i < lines.length; i++) {
  const colonIndex = lines[i].indexOf(':') // indexOf, не split — значение может содержать ':'
  const key = lines[i].slice(0, colonIndex).trim().toLowerCase() // HTTP headers case-insensitive
  const value = lines[i].slice(colonIndex + 1).trim()
  headers[key] = value
}
```

> ⚠️ Событие `'data'` на TCP-сокете может прийти **несколько раз** для одного HTTP-запроса. TCP не гарантирует, что весь запрос прилетит одним куском. Для учебного сервера это не проблема, но `http.createServer()` собирает чанки автоматически.

## 8. Заголовки, которые Backend-инженер обязан знать

### Заголовки запроса (читаешь)

| Header | Зачем | Пример |
|---|---|---|
| `Content-Type` | Формат тела запроса | `application/json`, `multipart/form-data` |
| `Authorization` | Токен аутентификации | `Bearer eyJ...` |
| `Accept` | Какой формат ответа хочет клиент | `application/json` |
| `Idempotency-Key` | Защита от дублей (см. раздел 11) | `uuid-v4-string` |
| `X-Request-Id` | Трассировка запроса через систему | `req_a1b2c3` |

### Заголовки ответа (выставляешь)

| Header | Зачем | Когда |
|---|---|---|
| `Content-Type` | Формат тела ответа | **Всегда** (кроме 204) |
| `Content-Length` | Длина тела **в байтах** | **Всегда** когда есть body |
| `Location` | URL созданного ресурса | После `201 Created` |
| `Retry-After` | Через сколько секунд повторить | При `429` или `503` |
| `Cache-Control` | Кэширование ответа | GET-эндпоинты |
| `X-Request-Id` | Эхо ID для дебага | **Всегда** — прод must-have |
| `Set-Cookie` | Установка куки | Auth-сценарии |
| `Allow` | Список допустимых методов | При `405 Method Not Allowed` |

**🔴 Важно про `Content-Length`:** Это длина в **байтах**, не в символах. `"Привет".length` = 6, но `Buffer.byteLength("Привет")` = 12 (UTF-8 кириллица = 2 байта на символ). Если Content-Length неправильный — клиент обрежет ответ или зависнет, ожидая ещё данных.

### Заголовок `Location` при `201 Created`

Когда бэкенд создаёт ресурс, заголовок `Location` указывает клиенту URL нового ресурса:

```http
POST /api/agents  →  HTTP/1.1 201 Created
                     Location: /api/agents/73
```

Клиент может сделать `GET /api/agents/73` чтобы получить полную сущность.

### Заголовок `Retry-After` при `429 Too Many Requests`

Говорит клиенту, сколько секунд ждать перед повторным запросом:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 30
```

> ⚠️ `Retry-After` может быть числом (секунды) или датой в формате HTTP-Date. На бэкенде безопаснее отдавать секунды.

## 9. Идемпотентность HTTP методов (Idempotency)

**Идемпотентность** — это важнейшее свойство HTTP-метода: сколько бы раз вы ни повторили один и тот же запрос, состояние сервера (данные в базе) не должно измениться после первого успешного выполнения. 

Это критично, потому что сеть может "моргнуть", и фронтенд (или мобильное приложение) отправит запрос повторно (Retry).

*   **GET (Идемпотентен):** Чтение списка пользователей. Можно прочитать хоть миллион раз — база данных не изменится.
*   **PUT (Идемпотентен):** Полная замена сущности по конкретному URL. `PUT /users/1 { name: "Alex" }`. Если отправить это 10 раз, в базе просто 10 раз сохранится "Alex".
*   **DELETE (Идемпотентен):** Удаление. `DELETE /users/1`. Первый запрос удалит пользователя. Остальные 9 запросов вернут ошибку `404 Not Found` (или `200 OK`), но логическое состояние системы не изменится — пользователь всё так же удален.
*   **POST (НЕ идемпотентен):** Создание новой сущности. `POST /payments { amount: 100 }`. Если из-за лагов сети этот запрос уйдет 5 раз, сервер спишет с карты 500 долларов. Это катастрофа.
*   **PATCH (Зависит от логики):** Частичное обновление.
    *   Идемпотентный пример: `PATCH /users/1 { name: "Alex" }`.
    *   НЕ идемпотентный пример: `PATCH /users/1 { $increment: { balance: 100 } }`. Повторы накинут лишние деньги.

## 10. PUT vs PATCH — полная замена vs частичное обновление

### Пример из Omnia

Агент в базе данных:

```json
{
  "id": 73,
  "name": "Sales Bot",
  "model": "gpt-4",
  "temperature": 0.7,
  "systemPrompt": "You are a sales assistant"
}
```

### PUT — Полная замена ресурса

PUT означает: «Вот ВЕСЬ ресурс. То, чего нет в запросе — не существует».

```http
PUT /api/agents/73

{ "name": "Support Bot" }
```

Результат в базе — **пропущенные поля обнулены**:

```json
{
  "id": 73,
  "name": "Support Bot",
  "model": null,
  "temperature": null,
  "systemPrompt": null
}
```

### PATCH — Частичное обновление

PATCH означает: «Измени только то, что я передал. Остальное не трогай».

```http
PATCH /api/agents/73

{ "name": "Support Bot" }
```

Результат в базе — **остальные поля не тронуты**:

```json
{
  "id": 73,
  "name": "Support Bot",
  "model": "gpt-4",
  "temperature": 0.7,
  "systemPrompt": "You are a sales assistant"
}
```

### Сравнительная таблица

| Критерий | PUT | PATCH |
|---|---|---|
| Что отправляешь | **Всю** сущность целиком | **Только** изменённые поля |
| Пропущенные поля | Обнуляются / удаляются | Остаются без изменений |
| Идемпотентность | ✅ Всегда | ⚠️ Зависит от реализации |
| Типичное использование | Полное редактирование формы | Инлайн-изменение одного поля |
| Пример в Omnia | Перезапись всех настроек агента | Переключение `isActive: false` |

### 🔴 Почему на практике в 2026 почти все используют PATCH

1. Фронтенд редко отправляет ВСЮ форму — обычно меняется 1-2 поля.
2. PUT требует сначала GET (получить текущее состояние) → лишний запрос.
3. PUT опасен при **lost update problem**: два пользователя одновременно делают GET, получают одно состояние. Первый отправляет PUT с изменённым `name`. Второй отправляет PUT с изменённым `model` — но `name` у него старый. Изменение первого пользователя потеряно.

> В Omnia: **PATCH** для 90% обновлений, **PUT** только для полной замены конфига (например, перезапись всего systemPrompt + параметров агента).

## 11. Защита от дублей: Паттерн Idempotency-Key

Как сделать так, чтобы повторный нестабильный `POST` запрос (например, оплата) не создал дубликат?
В индустрии используют заголовок **Idempotency-Key** (Ключ идемпотентности).

**Как это работает:**
1.  Фронтенд перед отправкой запроса генерирует уникальную строку — **UUID** (Universally Unique Identifier — Универсальный уникальный идентификатор).
2.  Добавляет его в заголовки: `Idempotency-Key: a1b2c3d4...`
3.  Бэкенд получает запрос и проверяет в базе данных (или в Redis): "Я уже видел ключ `a1b2c3d4`?".
4.  Если **нет**: Бэкенд выполняет бизнес-логику (списывает деньги), сохраняет в базу связь «Платеж выполнен для ключа a1b2c3d4» и отвечает `200 OK`.
5.  Если **да**: Бэкенд понимает, что это случайный дубль из-за плохой сети. Он **не выполняет** логику повторно, а просто достает из базы прошлый успешный результат и снова отвечает `200 OK`.

## 12. HTTP Status Codes (Коды состояний) — Подробный разбор для Backend


### 2xx (Success — Успех)

```
200 OK          — Стандартный ответ при успехе (GET, PUT, PATCH).
201 Created     — Ресурс создан (POST). ОБЯЗАТЕЛЬНО: заголовок Location с URL нового ресурса.
204 No Content  — Успешно, тело ответа отсутствует (DELETE). Без Content-Type, без body.
```

### 3xx (Redirection — Перенаправление)

```
301 Moved Permanently     — Ресурс навсегда перемещён на новый URI. Браузер закэширует это жёстко.
302 Found                 — Временный редирект (де-факто стандарт, но неоднозначный — может менять метод).
307 Temporary Redirect    — Временный редирект с гарантией сохранения метода (POST останется POST).
308 Permanent Redirect    — Как 301, но с гарантией сохранения метода.
304 Not Modified          — Ресурс не изменился с прошлого запроса (кэш актуален, тело не отправляется).
```

> 🔴 **Разница 301 vs 308:** При 301 браузер может сменить метод с POST на GET (историческое поведение). При 308 — метод гарантированно сохраняется. Для API-редиректов безопаснее 308.

### 4xx (Client Error — Ошибка клиента)

Здесь самое сложное — **выбрать правильный код**. Примеры из Omnia:

```
400 Bad Request           — Запрос синтаксически сломан.
                            Битый JSON, отсутствует обязательное поле, неправильный тип данных.
                            POST /agents body: "не JSON вообще{" → 400

401 Unauthorized          — Кто ты? Не аутентифицирован.
                            Токен отсутствует, протух, подпись невалидна.
                            GET /agents (без Authorization header) → 401

403 Forbidden             — Я знаю кто ты, но тебе НЕЛЬЗЯ.
                            Токен валиден, но роль не та.
                            DELETE /workspaces/1 (user.role = "viewer") → 403

404 Not Found             — Ресурс не существует.
                            GET /agents/99999 → 404

409 Conflict              — Бизнес-логика конфликтует с текущим состоянием.
                            POST /users { email: "alex@test.com" } (email уже в базе) → 409

422 Unprocessable Entity  — Синтаксис ОК, но данные семантически бессмысленны.
                            POST /agents { name: "Bot", model: "gpt-99" } (модели не существует) → 422

429 Too Many Requests     — Rate limit сработал. Обязательно: заголовок Retry-After.
```

### 🔴 Критическая разница: 400 vs 422

*   **400** — запрос **синтаксически** сломан (битый JSON, нет обязательного поля).
*   **422** — синтаксис ОК, JSON парсится, но данные **семантически** не имеют смысла для бизнес-логики (ссылка на несуществующую модель, невозможное сочетание полей).

### 🔴 Критическая разница: 401 vs 403

```
401 = "Кто ты?"      → Покажи мне токен. Фронт должен показать форму логина.
403 = "Я знаю кто ты, но нет" → У тебя роль viewer, нужен admin. Фронт показывает "Нет доступа".
```

Если перепутать: вернёшь `403` без токена — фронт не покажет логин. Вернёшь `401` без прав — фронт будет бесконечно рефрешить токен. **Реальный баг на проде.**

### 5xx (Server Error — Ошибка сервера / Виноват бэкендер)

```
500 Internal Server Error — Необработанное исключение в коде Node.js.
                            НИКОГДА не возвращай стектрейс клиенту — это дыра безопасности
                            (пути файлов, имена зависимостей, фрагменты SQL).
                            Клиенту: { "error": "Internal server error", "requestId": "..." }
                            Логи: полный стектрейс в Pino/Winston.

502 Bad Gateway           — Nginx жив, Node.js — нет (упал/перезагружается).
504 Gateway Timeout       — Node.js жив, но завис на тяжёлом запросе. Nginx не дождался.
```

### 🚫 Антипаттерн: `200 + { error: "..." }`

```javascript
// 🚫 ПЛОХО — фронтенд фильтрует по response.ok (коды 200-299)
res.status(200).json({ error: "User not found" })

// ✅ ХОРОШО — правильный HTTP-код
res.status(404).json({ error: "User not found" })
```

## 13. URL, Path Params, Query Params

### Правило REST: существительные во множественном числе = коллекции ресурсов

### Path Params (Параметры пути)

Используются для **идентификации конкретного ресурса** в коллекции. Без этого параметра запрос теряет смысл.

```
GET /api/workspaces/12/agents/5
                   ^^        ^
                   Path Params — уникальные ID
```

В Express:
```javascript
// Маршрут: /api/agents/:id
// Запрос:  GET /api/agents/42
req.params.id  // => "42" (СТРОКА, не число!)
```

**Правило:** Если ресурс не может существовать или быть найден без этого значения — это Path Param.

### Query Params (Параметры строки запроса)

Используются для **фильтрации, сортировки, пагинации или поиска** внутри коллекции. Они **опциональны** — без них вернётся вся коллекция.

```
GET /api/agents?status=active&limit=10&page=2
               ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
               Query Params — фильтрация, пагинация
```

В Express:
```javascript
// Запрос: GET /api/agents?status=active
req.query.status  // => "active"
```

### Сравнительная таблица

| Критерий | Path Params (`/agents/:id`) | Query Params (`/agents?status=active`) |
|---|---|---|
| **Назначение** | Идентификация конкретного объекта | Фильтрация, сортировка, пагинация, поиск |
| **Обязательность** | Обязателен (без него — другой эндпоинт) | Опционален (без него — вся коллекция) |
| **Пример в Omnia** | `GET /agents/73` (бот #73) | `GET /agents?model=gpt-4` (список GPT-4 ботов) |
| **Кэширование** | Кэшируется как отдельный URL | На CDN — зависит от настроек |

### 🔴 Подводные камни

**1. Парсинг типов — всегда String!**

И Path Params, и Query Params приходят как **строки**. `req.params.id` = `"42"`, не `42`.

```typescript
// 🚫 Drizzle/Prisma упадут — ожидают число, получают строку
const agent = await db.select().from(agents).where(eq(agents.id, req.params.id));

// ✅ Явное приведение + валидация
const agentId = parseInt(req.params.id, 10);
if (isNaN(agentId)) {
  return res.status(400).json({ error: "Invalid agent ID" });
}
```

**2. Порядок роутов в Express — статические перед динамическими**

```javascript
// 🚫 GET /agents/active попадёт сюда, :id станет "active"
app.get('/agents/:id', handler)
app.get('/agents/active', handler) // НИКОГДА не выполнится

// ✅ Сначала статические, потом динамические
app.get('/agents/active', handler)
app.get('/agents/:id', handler)
```

**3. URL Encoding**

Спецсимволы и кириллица кодируются: пробел → `%20`, кириллица → `%D0%...`. Express декодирует автоматически, но при ручном парсинге нужен `decodeURIComponent()`.

**4. Массивы в Query Params**

```
GET /api?tags=node&tags=react
```

Express распарсит как массив: `req.query.tags = ['node', 'react']`. Бэкенд должен быть готов к тому, что параметр может прийти и как строка, и как массив.

## 14. Иерархия серверов в Node.js: от TCP до NestJS

```
net.createServer()     ← Голый TCP. Получаешь сырые байты, сам парсишь текст.
  ↓
http.createServer()    ← Node парсит HTTP за тебя → даёт req.method, req.url, req.headers
  ↓
Express(app)           ← Добавляет роутинг, middleware, res.json()
  ↓
NestJS                 ← Добавляет DI, декораторы, модули
```

Каждый уровень — обёртка над предыдущим. `http.createServer()` внутри использует `net.createServer()`. Express внутри использует `http.createServer()`.

### `net.createServer()` — модуль `node:net`

Встроенный модуль Node.js для работы с **TCP-сокетами** (уровень L4). Создаёт TCP-сервер, который слушает порт и принимает входящие соединения. Каждое новое соединение — объект `socket` (двусторонний канал для чтения/записи сырых байтов).

```javascript
import * as net from 'node:net'

const server = net.createServer(socket => {
  // callback вызывается на КАЖДОЕ новое подключение
  // 5 браузеров = 5 вызовов с разными socket-объектами

  socket.on('data', data => {
    // 'data' — клиент прислал байты
    // data — Buffer (массив байтов), не строка
    console.log(data.toString()) // UTF-8 → текст
  })

  socket.write('ответ')  // Отправить байты клиенту
  socket.end()           // Закрыть соединение
})

server.listen(3000) // Начать слушать порт 3000
```

### Модуль `node:crypto`

Встроенный модуль для криптографии: шифрование, хэширование, генерация случайных значений.

*   `crypto.randomUUID()` — генерация UUID v4 (криптографически стойкий уникальный идентификатор).
*   🔴 **Никогда не использовать `Math.random()` для токенов/сессий/ID** — он предсказуем (PRNG). Злоумышленник может вычислить следующие значения и угнать чужие сессии. Только `crypto.randomBytes()` или `crypto.randomUUID()`.

## 🎯 Ключевые вопросы на собеседовании (по всем темам)

1.  **«Чем 401 отличается от 403?»** — 401 = не аутентифицирован (нет токена). 403 = аутентифицирован, но нет прав (роль не та).
2.  **«Что вернёшь на POST /users если email занят?»** — 409 Conflict. Не 400 (ввод валиден), не 500 (не баг).
3.  **«Чем 400 отличается от 422?»** — 400 = синтаксически сломан. 422 = синтаксис ОК, семантически бессмысленно.
4.  **«Зачем заголовок Location?»** — Возвращается с 201 Created. Содержит URL нового ресурса.
5.  **«Почему нельзя возвращать стектрейс в 500-ке?»** — Пути файлов, зависимости, фрагменты SQL = вектор атаки.
6.  **«Что будет если Node.js не ответил за 30 секунд, а перед ним Nginx?»** — Nginx вернёт 504. Node.js продолжит обработку (утечка ресурсов, если нет таймаута).
7.  **«Когда использовать Path Params, а когда Query Params?»** — Path = идентификация конкретного ресурса. Query = фильтрация/сортировка/пагинация коллекции.
8.  **«Что происходит внутри Express при `app.get('/users', handler)`?»** — Слушает TCP-сокет, парсит текст в объект req, прогоняет через middleware, сопоставляет URL с маршрутами, вызывает handler. `res.json()` = stringify + заголовки + запись в сокет.
9.  **«Чем `net.createServer` отличается от `http.createServer`?»** — `net` = сырые TCP-байты, сам парсишь. `http` = автоматический парсинг HTTP-протокола, готовые `req`/`res`.
10. **«Почему нельзя использовать `Math.random()` для генерации токенов?»** — Предсказуем (PRNG). Нужен CSPRNG: `crypto.randomBytes()` или `crypto.randomUUID()`.
